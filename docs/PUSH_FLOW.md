# Push Notification Flow

This document describes the complete push notification flow, from change detection to device delivery.

## Architecture

```
┌─────────────────┐      ┌──────────────────────┐      ┌─────────────┐
│  Blizzard API   │─────▶│  Elixir (Phoenix)    │─────▶│  Firebase   │
│  (WoW Status)   │      │  StatusChecker       │      │     FCM     │
└─────────────────┘      │  WeeklyResetWorker   │      └─────────────┘
                         └──────────────────────┘             │
                                                              ▼
                                                    ┌──────────────────┐
                                                    │  Mobile App      │
                                                    │  (Android + iOS) │
                                                    └──────────────────┘
```

## Notification Types

### 1. Status Change (Individual)

**Trigger:** A realm changes from UP to DOWN (or vice versa)

**Flow:**
1. `StatusChecker` detects change
2. Creates an event record in the database
3. Calls `Notifier.send_status_change/1`
4. `Notifier` fetches OAuth token via `goth` (service account)
5. POSTs to FCM v1 API → topic `wow_us_all_status`
6. FCM delivers to subscribed devices

**Payload:**

```json
{
  "message": {
    "topic": "wow_us_all_status",
    "notification": {
      "title": "Realm OFFLINE",
      "body": "Area 52 went OFFLINE • Americas & Oceania"
    },
    "data": {
      "type": "individual",
      "realm_name": "Area 52",
      "status": "OFFLINE"
    }
  }
}
```

### 2. Status Change (Aggregated)

**Trigger:** More than 5 realms change simultaneously

**Flow:**
1. `StatusChecker` detects multiple changes
2. Aggregates into a single notification
3. Sends summary instead of individual notifications

**Payload:**

```json
{
  "message": {
    "topic": "wow_us_all_status",
    "notification": {
      "title": "Status changes",
      "body": "12 realms changed (8 offline, 4 online)"
    },
    "data": {
      "type": "aggregated",
      "changes": "..."
    }
  }
}
```

### 3. Weekly Reset

**Trigger:** Oban cron job (every Tuesday at 15:00 UTC)

**Flow:**
1. Oban fires `WeeklyResetWorker`
2. Worker calls `Notifier.send_weekly_reset/1`
3. `Notifier` sends to topic `wow_us_weekly_reset`

**Payload:**

```json
{
  "message": {
    "topic": "wow_us_weekly_reset",
    "notification": {
      "title": "Weekly reset",
      "body": "US/Latin/Oceanic — Tuesdays 15:00 UTC"
    },
    "data": {
      "type": "weekly_reset",
      "region": "us",
      "timestamp": "2024-01-16T15:00:00Z"
    }
  }
}
```

## FCM Topics

| Topic | Description | Subscribe when |
|-------|-------------|----------------|
| `wow_us_all_status` | Realm status changes | "Status changes" toggle is on |
| `wow_us_weekly_reset` | Weekly reset reminder | "Weekly reset" toggle is on |

## FCM v1 API

The backend uses the **FCM HTTP v1 API** directly:

- **Endpoint:** `POST https://fcm.googleapis.com/v1/projects/{project_id}/messages:send`
- **Auth:** OAuth 2.0 Bearer token (generated by `goth` from the Firebase service account)
- **Project ID:** extracted automatically from `FIREBASE_SERVICE_ACCOUNT_JSON`

> Note: FCM `data` values must all be strings. The `Notifier` stringifies complex values automatically.

## Spam Prevention

### Deduplication (Flapping Prevention)
- **Window:** 5 minutes
- **Action:** Ignores rapid UP/DOWN oscillations within the window

### Aggregation
- **Threshold:** > 5 realms change simultaneously
- **Action:** Sends one aggregated notification instead of N individual ones

### Rate Limiting
- **Status changes:** Max 1 notification per 60 seconds
- **Weekly reset:** Once per week (enforced by Oban cron)

## Troubleshooting

### Notifications not arriving

1. **Check backend logs** for `Notifier` errors
2. **Verify FCM credentials:** `FIREBASE_SERVICE_ACCOUNT_JSON` must be valid
3. **Check topic subscription** in the mobile app
4. **Verify permissions** were granted on the device

### Test manually
